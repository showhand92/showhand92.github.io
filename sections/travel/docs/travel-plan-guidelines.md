# 旅行计划页面规范

这份文档只记录旅行计划相关的内容、目录、数据、样式和交互规则。站点入口、主页、Blog 等非旅行结构见 `sections/docs/site-structure.md`。

## 目录边界

- 旅行计划总入口是 `/sections/travel/`。
- 具体旅行计划放在 `sections/travel/plans/<trip-slug>/`，例如 `plans/kyushu/`、`plans/europe/`。
- 每个完整旅行计划通常包含：
  - `index.html`：行程计划。
  - `photos.html`：照片与美食。
  - `map.html`：交互地图。
  - `itinerary-data.js`：行程页、照片页和地图页共用的数据。
- 旅行公共资源放在：
  - `sections/travel/css/`
  - `sections/travel/js/`
  - `sections/travel/docs/`
- 旅行详情页不要依赖 `home.css` 或 `sections/site-index.css`。

## 页面入口

- Travel 总页只展示旅行计划入口，不承载具体行程正文。
- 旅行计划卡片要整体可点击，不只标题可点。
- 具体旅行计划内部顶部选项卡顺序固定为：
  1. `行程计划`
  2. `照片与美食`
  3. `交互地图`
  4. `Travel`
- 当前页高亮；`Travel` 返回 `/sections/travel/`。
- 具体旅行计划内部不要把 Home 直接带回个人主页。

## 共享数据

- `itinerary-data.js` 是行程页、照片页和地图页的共同来源。
- 数据挂到 `window.TravelItineraries.<trip-slug>` 下，至少包含：
  - `days`
  - `points`
  - `routeSegments`
- `days` 是统一天数列表，用于行程页、地图页和目录。
- `points` 是每天时间表、地图点位、照片标签和弹窗内容的来源。
- `routeSegments` 是地图路线和左侧交通摘要的来源。
- 不要在 `index.html` 和 `map.html` 分别手写两套时间/安排数据。静态 HTML 可以保留兜底内容，但实际渲染应以共享数据为准。

### 点位字段

- `id`：稳定编号，例如 `D3-4`。
- `day`：数字天数。
- `kind`：`transit` 或 `visit`。
- `time`：单时刻或时间段。
- `name`、`note`、`lat`、`lng`。
- 可选 `featured`、`photo`、`photoSource`。

### 点位类型

- `transit`：到达就离开、出发、抵达、换乘、机场、车站、码头等交通节点。时间只标记一个时刻，例如 `10:45`。
- `visit`：景点、街区、用餐、观景、购物、休整等停留节点。时间标记一段时间，例如 `10:30-11:30`。
- 不要用一个点位同时表达交通耗时和景点停留。交通耗时放在 `routeSegments.routeTag`。

## 行程页

- 行程正文默认不用具体日期，用 `D1`、`D2`、`D3` 等相对天数。
- 每一天要接近可执行时间线，不只写城市跳转或景点列表。
- 长线计划通常每天 4-8 个核心节点比较合适。
- 如果原始资料里有更细的时间规划，行程页和地图页都要体现。
- 每天行程卡片建议包含：
  - 当天标题，例如 `D3｜城市 / 区域 / 主题`。
  - 标签行，用于标记交通、门票、景观、机动安排等。
  - 时间表，第一列是时间，第二列是具体安排。
  - `吃饭建议`，说明午餐、晚餐或当天适合的餐饮区域。
- `吃饭建议` 写餐饮方向和取舍，不写成餐厅排名；如果提到和牛、寿喜烧、正式锅物、Fine dining 等高预算选项，要标为可选升级餐，并说明适合放在哪一天、是否会抬高预算。
- 预算表里的 `餐饮` 上限要覆盖可选升级餐；如果升级餐不是必吃，说明普通餐饮低位预算和升级餐高位预算的区别。
- 页面前部保留 `出发前 Checklist`，用于机票、住宿、交通 App、证件、票券、预约项目等出发前确认。
- 页面后部保留 `住宿、交通、吃饭和预算`。
- 如果路线有关键预订、风险或调整原则，可以在预算前增加 `关键预订与票务`、`风险与调整原则` 等独立 section。

## 照片与美食页

- 照片页标题使用 `<目的地>照片与美食清单` 或 `照片与美食清单`。
- 照片页顶部选项卡与行程页、地图页一致。
- 照片页应覆盖核心景点、关键体验点和行程页明确推荐的餐饮方向。
- 照片页应该是地图照片点的超集：`itinerary-data.js` 里有 `photo` 的点，照片页必须能展示。
- 可以额外加入本次可能不去、但有助于取舍和理解路线的备选地点照片；也可以加入不绑定地图点的美食照片。
- 照片分组应按地点或国家/区域组织，不要把明显不同区域混在一起。
- 每个照片卡片至少包含：
  - 景点、区域或美食名称。
  - 图片。
  - 简短说明。
  - 图片来源与许可链接。
- 外部照片必须标注来源，地图弹窗和照片页都应提供来源链接。
- 图片优先选择清晰、能准确代表景点的照片；无法加载或太模糊时需要替换。
- 不要把多个独立景点硬塞进一个照片卡片，除非它们确实是同一观景点、同一建筑群或同一段路线。
- 美食照片是行程页吃饭建议的视觉补充，不写成餐厅排名；如果照片不是指定餐厅出品，说明只表达菜品或餐饮方向。

## 交互地图

- 地图页标题使用 `<目的地>行程地图` 或 `行程地图`。
- 地图页顶部选项卡样式必须和行程页、照片页一致。
- 桌面端左侧为每天详情，右侧为地图；滚动左侧详情时，地图区域不跟着上下滚动。
- 左侧宽度可拖动调整。
- 移动端改为上下布局，保证地图有稳定高度。
- 地图图层控制默认折叠。
- 默认状态：
  - 不选中任何一天时，所有路线正常显示。
  - 不显示地图标签。
  - 路线颜色可以稍淡，但仍要有足够可见度。
- 选中某一天时：
  - 该天路线明显高亮。
  - 其他天路线保留可见，但降低透明度，建议约 `0.5`。
  - 只显示被选中当天的地图标签。
  - 点击同一天可取消选中，回到默认状态。
- 点击左侧行程项时：
  - 地图飞到对应点。
  - 打开对应弹窗。
  - 如果有照片，弹窗显示照片和图片来源与许可。
- 左侧行程项如果对应点位有 `photo`，自动显示 `照片` 标签；没有 `photo` 不显示。
- 标签避让优先按当天计算。默认不显示标签，所以跨天相同坐标不应影响当天标签布局。

### 路线与交通摘要

- `routeSegments` 只描述移动路线，不描述景点停留。
- `label` 写清哪几个点之间，例如 `D2-1 Interlaken Ost -> D2-4 少女峰`。
- `routeTag` 写交通方式、耗时、票务或费用，例如 `上山约2小时 / 需票`、`渡轮约15分 / ¥250`、`SBB约3小时 / Pass`。
- 交通摘要应按路线逐行显示，不要把多段交通挤在同一行。
- 城市内步行线可以作为 `routeSegments` 保留；如果没有明确耗时或费用，可以不写 `routeTag`。
- 点位增加后，路线线条也要同步更新到新的关键点，不要保留旧的粗略 2-3 点路线。
- 地图图例使用 `TravelMap.renderLegend(...)` 生成，页面只保留 `<div id="map-legend"></div>` 占位和交通类型配置。
- 图例线条统一使用黑色，只通过线型提示差异；文案只写交通类型，不写“实线/虚线/点线”。
- 步行归入 `city` 类，与市内游览、接驳近似路线放在一起。

## 文案与预算

- 页面内容直接服务旅行使用，不要放太多解释“这个页面如何展示”的说明文字。
- 避免重复总述。路线主线、取舍原则、预算口径等信息只在真正需要的位置出现一次。
- 标题和副标题保持简洁，避免把天数、夜数、所有城市都堆进主标题。
- 行程页标题使用旅行计划名称，例如 `<目的地>旅行计划`。
- 每个页面标题下保留一句简短副标题，说明路线主线或页面用途。
- 预算给出大致量级，不需要精确到每一餐。
- 预算表至少包含住宿、城际交通、市内交通、餐饮、门票或活动、杂费、合计。
- 如果目的地使用外币，预算需要同时给当地货币和人民币粗算。
- 预算说明要注明估算口径，例如是否包含国际机票、购物、住宿是否两人分摊、汇率是否只是粗略折算。
- 预算区间应反映真实波动来源，例如旺季住宿、交通票策略、热门活动门票、是否多人分摊房费。

## 视觉与排版

- 字体使用统一旅行页字体：`"Avenir Next", "Segoe UI", "Microsoft YaHei", sans-serif`。
- 整体使用温和、清晰的暖色调，但不要让页面变成单一棕色、橙色或过度相近的色块。
- 行程、照片、地图页面的顶部选项卡样式要统一。
- 行程卡片、表格、照片卡片、地图侧栏的间距、字号和圆角应保持同一套设计语言。
- 时间列应便于扫描：时间字体加粗，数字宽度稳定，时间列和安排列间隙稳定。
- 表格在移动端必须可读：
  - 小屏下表格变成卡片式布局。
  - 每个单元格显示字段标签，例如 `时间`、`安排`、`项目`、`说明`。
  - 长文本允许换行，不能挤压或重叠。
- 多列表格需要给合理列宽。长项目名允许换行，不能和旁边列重叠。
- 行程页和照片页右下角保留：
  - `目录`：放在 `Top` 上方，跳转到每天、照片分组、预算、地图等关键区域。
  - `Top`：回到顶部。

## 颜色规则

- 每一天或每个城市段需要有稳定颜色。
- 同一城市或同一区域连续天数使用同一大色组内的色相偏移，变化要能看出来，不要只通过明暗强弱区分。
- 不同城市段使用可区分的色相组。
- 行程页卡片边线、交互地图路线、地图标签、左侧行程卡片应尽量共用同一套 day color。
- 颜色用于帮助理解路线，不要只作为装饰。

## CSS 与 JS

- 旅行详情页按类型引入对应 CSS：
  - `plans/<trip-slug>/index.html` 引入 `../../css/travel-common.css` 和 `../../css/travel-plan.css`。
  - `plans/<trip-slug>/photos.html` 引入 `../../css/travel-common.css` 和 `../../css/travel-photos.css`。
  - `plans/<trip-slug>/map.html` 引入 `../../css/travel-common.css` 和 `../../css/travel-map.css`。
- 公共 CSS 分工：
  - `travel-common.css`：通用字体、背景、header、顶部选项卡、`Top` 和 `目录`。
  - `travel-plan.css`：行程卡片、行程表、预算表、移动端表格。
  - `travel-photos.css`：照片与美食网格、卡片、图片来源与许可、优先级表。
  - `travel-map.css`：地图布局、侧栏、图例、弹窗、照片标签。
- 目的地页面可以保留少量局部 `<style>`，只用于目的地颜色变量、day color、个别页面特例或过渡样式。
- 如果两个及以上旅行计划会共享某个样式，优先放进对应的 `travel-*.css`。
- 公共 JS 都是浏览器端脚本，不是 Node 后端程序，也不需要手动运行。
- 行程页加载顺序：
  - `./itinerary-data.js`
  - `../../js/travel-plan-render.js`
  - `TravelPlanRenderer.renderSchedule("<trip-slug>")`
- 照片页加载顺序：
  - `./itinerary-data.js`
  - `../../js/travel-photos-render.js`
  - `TravelPhotosRenderer.renderMapPhotoSuperset("<trip-slug>")`
- 地图页加载顺序：
  - Leaflet JS
  - `./itinerary-data.js`
  - `../../js/travel-map.js`
  - 页面内初始化底图、`routeStyles`、`dayColors` 等配置
  - `TravelMap.renderLegend(...)`
  - `TravelMap.createItineraryMap({...})`
- 不要在每个旅行计划里复制 marker、route、highlight、popup、legend 这些公共地图逻辑。
- 修改 `travel-common.css` 时要特别检查地图页：普通页面的 `main` 宽度和 padding 不能影响 `<main class="layout">`。地图页 `.layout` 必须保持 `width: 100%`、`max-width: none`、`margin: 0`、`padding: 0`。

## 新增或更新流程

- 先确定目录名，例如 `sections/travel/plans/<trip-slug>/`。
- 根据原始资料整理每天时间线、核心景点、交通节点、住宿区域、餐饮建议、票务和预算。
- 如果从 Word、Markdown、HTML 或其他资料生成计划，需要读取细节时间安排，不能只做粗略总结。
- 先生成或更新 `itinerary-data.js`，再同步 `index.html`、`photos.html` 和 `map.html`。
- 新增地图时，先定义每天点位和路线，再同步左侧行程列表、照片标签和颜色。
- 新增点位时：
  - 先按行程页时间线抽取点。
  - 交通点用单时刻。
  - 停留点用时间段。
  - 一天点位太少时，回到原始资料补充具体景点、换乘、用餐或休整节点。
  - 一天点位过多时，只保留对路线理解有帮助的节点，避免地图标签过密。
- 知名景点或核心体验点用 `featured: true` 标记；标记为 `featured` 的点原则上必须同时提供 `photo` 和 `photoSource`。
- 修改一个旅行计划的通用交互或样式时，要检查其他旅行计划是否也应同步，避免不同目的地格式分裂。

## 检查清单

- 路径检查：
  - 旅行计划在 `sections/travel/plans/<trip-slug>/`。
  - CSS 引用使用 `../../css/...`。
  - JS 引用使用 `../../js/...`。
  - `Travel` 链接返回 `/sections/travel/`。
- 数据检查：
  - `days`、`points`、`routeSegments` 同步。
  - 行程页和地图页时间安排一致。
  - 有 `photo` 的点在地图弹窗和照片页都能显示来源。
  - `featured` 点有照片和来源。
- 交互检查：
  - 默认不显示地图标签。
  - 选中某一天后只显示当天标签。
  - 点击左侧行程项能飞到地图点并打开弹窗。
  - 目录在 `Top` 上方。
- 视觉检查：
  - 行程页、照片页、地图页顶部选项卡一致。
  - 移动端表格不重叠。
  - 预算表长文本能换行。
  - 地图页桌面端左侧滚动不带动右侧地图。
- 命令检查：
  - `node --check sections/travel/plans/<trip-slug>/itinerary-data.js`
  - `node --check sections/travel/js/travel-plan-render.js`
  - `node --check sections/travel/js/travel-map.js`
  - `node --check sections/travel/js/travel-photos-render.js`
  - `git diff --check`
